# P1.2 CI Gate — TDD Test Plan

## Context

The current pipeline has two gaps:

1. `apps/myapps/` deploy on push to `main` (`.github/workflows/myapps-prod-deploy.yml`) has **no test-passing precondition** — a red unit or E2E run does not block the deploy workflow from running in parallel.
2. Required status checks are enforced on `main` (PR #143), but the deploy workflow is not gated on those checks — it fires off on push regardless.

The P1.2 CI gate closes both gaps: required tests must be green on the merge commit before any deploy workflow starts, and every gated failure mode must have an xfail test that fails before the gate is wired and passes after.

This plan is the xfail catalog Vi will use (P1.2-E). Vi writes the tests; she does not implement the gate (P1.2-F). Each test entry below is precise enough for Vi to execute without clarification.

## Failure modes the gate must pin

Each row is a distinct failure mode the gate must prevent. The gate is considered correctly wired when every xfail test below is green.

| # | Failure mode | Current behaviour (pre-gate) | Required behaviour (post-gate) |
|---|---|---|---|
| F1 | Unit tests red on PR → PR mergeable | PR may merge if reviewer approves; `myapps-test.yml` unit job is not a required check | PR merge blocked by required status check until unit job is green |
| F2 | E2E tests red on PR → PR mergeable | Same as F1 for E2E job | Same as F1 for E2E job |
| F3 | Push to `main` with red unit tests triggers deploy | `myapps-prod-deploy.yml` fires on push to `main`; there is no dependency on `myapps-test.yml` result for the same SHA | Deploy workflow must either (a) not start, or (b) start and abort, when unit tests for the same SHA are red |
| F4 | Push to `main` with red E2E triggers deploy | Same as F3 for E2E | Same as F3 for E2E |
| F5 | Workflow file deleted/renamed sidesteps required check | Required status check referencing a workflow name that no longer exists silently becomes non-blocking | Presence and name of each required check workflow is verifiable in repo; removing or renaming it without updating branch-protection config fails a repo-invariant test |
| F6 | TDD-Waiver misuse — `chore:` commit touching `apps/myapps/**` skips xfail-first via waiver | `tdd-gate.yml` honours `TDD-Waiver:` trailer unconditionally | Gate invariant documented + test that the waiver trailer is only honoured under declared conditions (PR #143 landed branch protection; confirm waiver semantics did not regress) |
| F7 | Scope-validation bypass — `apps/**` diff committed with `chore:` prefix | Pre-commit scope hook rejects locally; CI has no equivalent | Server-side CI check rejects the same cases so a pushed commit with bad prefix fails a required check |
| F8 | Deploy workflow runs but target project is wrong (e.g. `PROJECT_ID` var unset) | Deploy runs against whichever project `vars.FIREBASE_PROJECT_ID` resolves to, including empty | Gate verifies required repo/environment vars are set before deploy starts |
| F9 | Lint red on PR → PR mergeable / deploy fires | `.github/workflows/ci.yml:28` runs `turbo run lint` but the lint job is not a required check for `main`; deploy does not depend on it | Lint job is a required status check on `main`; deploy workflow will not start with red lint on HEAD |
| F10 | Build red on PR → PR mergeable / deploy fires | `.github/workflows/ci.yml:31` runs `turbo run build` but the build job is not a required check; deploy has its own internal `npm run build` step but that doesn't prevent the deploy workflow from starting | Build job is a required status check on `main`; deploy workflow will not start with red build on HEAD |

F1–F4, F9, F10 are the **core gate** — the others are invariants the gate depends on and must not silently regress.

## xfail test catalog

Each entry below is one test Vi must author in P1.2-E. All tests live in the repo so they run in CI as part of the gate they're pinning. Initial expected result: **fails** (xfail). Post-gate expected result: **passes**.

Commit the xfail tests with a `TDD-Waiver: p1-2-ci-gate` trailer is NOT permitted — the whole point of P1.2 is the gate, so the gate's own xfail tests must be real xfail entries, not waivers. The `tdd-gate.yml` workflow should be temporarily tolerant during P1.2-E (tests committed as `test.fail(...)` / `it.failing(...)`) so the xfail commits themselves pass the xfail-first check.

Framework: **Vitest** for logic tests (matches ADR §1 decision and P1.4 harness). **Bash + `shellcheck` + a thin Vitest runner** (`describe('branch-protection invariants', …)`) for GitHub-API and repo-config assertions — same harness as existing repo-invariant tests under `apps/myapps/functions/src/__tests__/` or a new `tests/ci-gate/` directory (Vi picks, but must be inside a package that `scripts/test-all.sh` picks up).

### T1 — Unit-test-red PR is not mergeable (pins F1)

- **Assertion:** Given a PR where `myapps-test.yml` unit job reported failure on the head SHA, `gh api repos/:owner/:repo/pulls/<n>` returns `mergeable_state != "clean"` AND `gh api repos/:owner/:repo/commits/<sha>/status` aggregates to `failure`. A helper that simulates this (fixture PR or mocked GitHub API response) asserts: attempting `gh pr merge` returns a non-zero exit with a "required status check failed" message.
- **File path:** `tests/ci-gate/required-checks.test.ts` (new).
- **Failure mode it pins:** F1.
- **Initial expected result:** fails — because `myapps-test.yml` is not yet declared as a required check in branch protection config (P1.2-F will declare it).
- **Post-gate expected result:** passes — the workflow name `MyApps — Tests (unit + E2E) / Unit tests (Vitest)` is in the required-checks list of `main` branch protection, fetched via `gh api`.

### T2 — E2E-red PR is not mergeable (pins F2)

- **Assertion:** Identical shape to T1 but for the `MyApps — Tests (unit + E2E) / E2E tests (Playwright / Chromium)` job name.
- **File path:** `tests/ci-gate/required-checks.test.ts` (same file, second test case).
- **Failure mode it pins:** F2.
- **Initial expected result:** fails — E2E job name not in required-checks list.
- **Post-gate expected result:** passes.

### T3 — Deploy workflow refuses to start on red head (pins F3 + F4)

- **Assertion:** `.github/workflows/myapps-prod-deploy.yml` declares either (a) `needs: [test-unit, test-e2e]` with the test jobs promoted into the same workflow file, or (b) a `workflow_run` trigger on `MyApps — Tests (unit + E2E)` with `conclusion: success` filter, or (c) an initial in-job `gh api` precheck step that fetches the combined status for `github.sha` and exits non-zero on `failure`/`pending`. The test parses the YAML and asserts one of the three patterns is present; it does NOT prescribe which pattern (P1.2-F chooses).
- **File path:** `tests/ci-gate/deploy-workflow-gated.test.ts` (new).
- **Failure mode it pins:** F3, F4.
- **Initial expected result:** fails — current `myapps-prod-deploy.yml` (lines 3–11 trigger block) has an unconditional `on: push: branches: [main]` and no gating on test results.
- **Post-gate expected result:** passes.

### T4 — Simulated red test run on main does not produce a successful deploy run (pins F3 + F4, integration-level)

- **Assertion:** Fixture/mocked: given a commit SHA on `main` where `combined status == failure`, a dry-run of the deploy workflow's precheck step exits non-zero. Implemented as a Vitest test that shells out to `bash scripts/deploy/preflight-check.sh <sha>` (a helper Vi/Jayce must introduce in P1.2-F if pattern (c) above is chosen) OR as a workflow-yaml contract test (if pattern (a)/(b) is chosen, this test collapses into T3 and is skipped).
- **File path:** `tests/ci-gate/deploy-preflight.test.ts` (new; conditionally skip per pattern).
- **Failure mode it pins:** F3, F4 (runtime enforcement, not just declarative).
- **Initial expected result:** fails — no preflight script and no `needs:`/`workflow_run` wiring exists.
- **Post-gate expected result:** passes OR is legitimately skipped if pattern (a)/(b) was chosen in P1.2-F (T3 covers the declarative side either way).

### T5 — Required workflow names are present and spelled correctly (pins F5)

- **Assertion:** `tests/ci-gate/required-checks.test.ts` fetches `gh api repos/:owner/:repo/branches/main/protection/required_status_checks/contexts` and asserts every listed context name corresponds to a job that exists in some `.github/workflows/*.yml` file (parsed at test time). A listed context with no matching `jobs.<id>.name` fails the test — catches both deletion and rename drift.
- **File path:** `tests/ci-gate/required-checks.test.ts` (third test case).
- **Failure mode it pins:** F5.
- **Initial expected result:** fails — at minimum because the required-contexts list does not yet include the two myapps-test job names.
- **Post-gate expected result:** passes after P1.2-F updates branch protection AND the myapps-test job names stay stable.

### T6 — TDD-Waiver trailer is honoured only for intended cases (pins F6)

- **Assertion:** Given a commit message with `TDD-Waiver: <reason>` trailer, the `tdd-gate.yml` check skips the xfail-first rule only if `<reason>` is non-empty AND the commit diff does not touch `apps/myapps/functions/src/**` production code. Unit test parses `tdd-gate.yml`'s bash logic (extracted into a sourced helper, or inline regex matched) and asserts the reason-required semantic.
- **File path:** `tests/ci-gate/tdd-waiver-semantics.test.ts` (new).
- **Failure mode it pins:** F6.
- **Initial expected result:** fails — inspect `.github/workflows/tdd-gate.yml:66` which currently grants the waiver unconditionally on trailer presence, ignoring the reason body.
- **Post-gate expected result:** passes after P1.2-F tightens the waiver check (or explicitly decides unconditional waiver is acceptable and this test is renamed to assert that — but the xfail forces an explicit decision instead of silent drift).

### T7 — Commit-scope rule enforced in CI (pins F7)

- **Assertion:** `validate-scope.yml` exists, runs on `pull_request`, and rejects `apps/**` diffs with `chore:`/`ops:` prefix and non-`apps/**` diffs with `feat:`/`fix:`/`perf:`/`refactor:` prefix, matching the pre-commit hook logic. Test parses the workflow and asserts both directions are covered by invoking the hook script in a fixture shell.
- **File path:** `tests/ci-gate/scope-validation-ci.test.ts` (new).
- **Failure mode it pins:** F7.
- **Initial expected result:** fails if `validate-scope.yml` lacks one of the two directions — confirm by reading it; if both are already covered, rewrite this test as an xfail asserting the workflow is a **required** status check (which it is likely not yet).
- **Post-gate expected result:** passes.

### T8 — Deploy workflow refuses to start when required vars missing (pins F8)

- **Assertion:** `myapps-prod-deploy.yml` includes an early step that exits non-zero when `vars.FIREBASE_PROJECT_ID` is empty or when any of the seven `VITE_FIREBASE_*` secrets referenced at deploy time are empty. Test parses the workflow and asserts the presence of a `Check required vars` step with the right failure conditions.
- **File path:** `tests/ci-gate/deploy-required-vars.test.ts` (new).
- **Failure mode it pins:** F8.
- **Initial expected result:** fails — current workflow has no var-presence preflight (uses the values directly; missing would manifest as a runtime deploy error, not an early abort).
- **Post-gate expected result:** passes.

### T9 — Lint-red PR is not mergeable; red lint on HEAD blocks deploy (pins F9)

- **Assertion:** Two cases in one file. (a) `gh api repos/:owner/:repo/branches/main/protection/required_status_checks/contexts` includes the lint job name from `.github/workflows/ci.yml` (current job name — Vi greps the file at test time rather than hardcoding). (b) The same workflow-YAML parser from T3 confirms `myapps-prod-deploy.yml` gates on lint via one of the three patterns (needs/workflow_run/preflight).
- **File path:** `tests/ci-gate/lint-gate.test.ts` (new).
- **Failure mode it pins:** F9.
- **Initial expected result:** fails — lint is not a required check and deploy does not gate on it.
- **Post-gate expected result:** passes.

### T10 — Build-red PR is not mergeable; red build on HEAD blocks deploy (pins F10)

- **Assertion:** Symmetric to T9 but for the build job in `ci.yml` (`turbo run build`, line 31).
- **File path:** `tests/ci-gate/build-gate.test.ts` (new).
- **Failure mode it pins:** F10.
- **Initial expected result:** fails — build is not a required check and deploy does not gate on it.
- **Post-gate expected result:** passes.

## Handoff notes for Vi (P1.2-E)

- All ten tests go in a new directory `tests/ci-gate/` (or co-located per team decision). Each is a Vitest file. Import GitHub API helpers via `@octokit/rest` with a `GITHUB_TOKEN` env var, or shell out to `gh api` — Vi's call.
- Commit these ten xfail tests in a single commit (or 2–3 grouped commits) with `chore:` prefix. They touch `tests/ci-gate/**` only — no `apps/**` diff, so `chore:` is correct.
- The tests reference runtime behaviour (branch protection, workflow files) — they must be runnable in CI, so `scripts/test-all.sh` (P1.7) or an equivalent invocation must pick up the `tests/ci-gate/` directory. Add a `package.json` entry if needed.
- All ten MUST fail on first commit. If any passes pre-gate, the test is wrong — rework it so it truly pins the failure mode.
- P1.2-F (implementation) turns them green one at a time. T3 and T4 may collapse depending on the chosen gating pattern — that's an expected design decision at P1.2-F time and is not a regression.
