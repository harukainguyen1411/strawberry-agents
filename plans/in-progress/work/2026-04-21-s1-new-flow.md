## 1. Context

S1 (`company-os/tools/demo-studio-v3/`) is the FastAPI backend + UI host for Demo Studio v3. <!-- orianna: ok -->
The prior flow propagated brand/market/content from the Slack slash-command through to session creation and exposed an `/approve` route for explicit human sign-off before build. Both behaviors are now invalid:

- The managed agent is the sole driver of config state (§1 desired flow in god plan).
- Brand/market/content must not be pre-filled from the Slack trigger; the session starts empty.
- `/approve` has already been deleted server-side; any remaining UI call-sites are dead weight.
- S1 used to render its own in-process preview; that is replaced by an iframe pointing at S5.
- Build progress and S4 verification results were surfaced by separate polling; they are replaced by a single SSE endpoint `/session/{id}/logs`.

This ADR covers all S1-side changes required to realize the desired E2E flow described in the god plan `plans/proposed/work/2026-04-21-demo-studio-v3-e2e-ship-v2.md` §1–§2.4. <!-- orianna: ok -->

## 2. Decision

Nine surfaces change inside `company-os/tools/demo-studio-v3/` (and one slack-relay-side PR is a hard dependency called out below). <!-- orianna: ok -->
All changes land in a single feature branch via sequenced PRs keyed to phases A–G below.
The implementation is TDD-first: xfail tests precede every impl commit per Rule 12.

### Surface inventory

1. **Slack trigger contract** — `POST /session/new` schema relaxed to accept `initialContext: {}`. <!-- orianna: ok -->
   Slack-relay side: separate PR in the slack-relay repo to drop brand/market/content from the POST body. <!-- orianna: ok -->
2. **`/approve` dead route audit** — confirm zero UI call-sites; remove any residual wiring in JS/TS frontend.
3. **Iframe S5** — replace deleted in-process preview with `<iframe src="{S5_BASE}/v1/preview/{sessionId}">`. <!-- orianna: ok -->
   Add **Open in fullview** button opening `{S5_BASE}/v1/preview/{sessionId}/fullview` in a new tab. <!-- orianna: ok -->
4. **Session doc schema migration** — extend `session_store.py` with four new fields; null defaults; migration script. <!-- orianna: ok -->
5. **projectId capture** — on `POST /session/{id}/build`, read S3's `projectId` from response, persist; pass stored value on subsequent builds. <!-- orianna: ok -->
6. **SSE logs endpoint** — `GET /session/{id}/logs` multiplexes S3 build events and S4 verification poll results. <!-- orianna: ok -->
7. **S4 polling loop** — background task per active session; writes terminal state to session doc; emits to SSE.
8. **Iterate UX** — re-enable chat + Build button on terminal verification; same `POST /session/{id}/build` endpoint is idempotent-by-projectId. <!-- orianna: ok -->
9. **MCP resource `get_last_verification`** — add to in-process MCP server so the managed agent can read the latest verification report and propose fixes (§6 Q5 pick (b) from god plan).

## 3. Phase breakdown

### Phase A — `/session/new` contract + slack-relay dependency

Remove brand/market/content from S1's `SessionCreateRequest` Pydantic model. Accept `initialContext: {}` (empty dict) as valid. <!-- orianna: ok -->
Document the slack-relay PR dependency in the PR description.

### Phase B — `/approve` dead-route audit and UI removal

Grep S1's frontend source for any `fetch`/`axios`/`XHR` calls to `/approve`. <!-- orianna: ok -->
Remove any found call-sites, route declarations, and UI button wiring. No server-side route change needed (already deleted). Update any dead type references.

### Phase C — Iframe S5 + fullview button

Replace S1's deleted in-process preview route reference with an `<iframe>` element pointing at `{S5_BASE}/v1/preview/{sessionId}`. <!-- orianna: ok -->
Add an **Open in fullview** anchor/button rendering `target="_blank"` to `{S5_BASE}/v1/preview/{sessionId}/fullview`. <!-- orianna: ok -->
Gate both on `S5_BASE` env var; if unset, render a placeholder.

### Phase D — Session doc schema migration

Extend `session_store.py` with: <!-- orianna: ok -->

```
projectId: str | None = None
verificationStatus: Literal['pending', 'passed', 'failed'] | None = None
verificationReport: dict | None = None
lastBuildAt: datetime | None = None
```

Write a migration script (`scripts/migrate_session_docs.py`) <!-- orianna: ok --> that reads all existing Firestore documents in the `demo-studio-sessions` collection and writes the four new fields with null defaults if absent. Add a dry-run flag. Migration is idempotent.

### Phase E — projectId capture on build

On `POST /session/{id}/build`: <!-- orianna: ok -->

1. Read `projectId` from the session doc (if set, pass it in the S3 request body).
2. Parse S3's response for `projectId`; persist via `session_store` setter.
3. Set `lastBuildAt` to `datetime.utcnow()`.
4. Respect SE kwarg-only `transition_status` contract — do not call `transition_status` with positional args.

### Phase F — SSE logs endpoint

`GET /session/{id}/logs` — StreamingResponse using `text/event-stream` content type. <!-- orianna: ok -->
Auth: reuse S1's existing session cookie (god plan §6 Q1 pick (a)).
Multiplex strategy: async generator that:

1. Subscribes to S3 build events (pull from `GET /build/{buildId}/events` SSE on S3, forwarded verbatim with `event: build` prefix).
2. Concurrently yields S4 poll results written by the Phase G background task via an in-process asyncio queue per session.

Single client subscription per session; second subscriber receives a 409 or is queued.

### Phase G — S4 polling background task

On session build start, spawn an `asyncio` background task (tracked on the session object) that:

1. Polls S4 `GET /verify/{projectId}` at a fixed interval (5 s default; configurable). <!-- orianna: ok -->
2. On terminal state (`passed` or `failed`): writes `verificationStatus` + `verificationReport` to session doc; emits a final SSE event; cancels the poll loop.
3. On non-terminal state: emits an in-progress SSE event; continues polling.
4. Timeout guard: if no terminal after `VERIFICATION_POLL_TIMEOUT_S` (default 300 s), write `failed` with a timeout report.

At most one poller per session; idempotent start (second call is a no-op if already running).

### Phase H — Iterate UX + idempotent build

When the SSE stream delivers a terminal `verificationStatus` event:

- Re-enable the chat input and **Build** button in the UI.
- Surface the `verificationReport` summary in the UI (collapsible panel or inline alert).

`POST /session/{id}/build` is already idempotent-by-projectId in the S3 contract (Phase E passes stored `projectId`). <!-- orianna: ok -->
No separate iterate endpoint is introduced.

### Phase I — MCP resource `get_last_verification`

Add a new MCP tool/resource `get_last_verification` to S1's in-process MCP server (post-MCP-merge). <!-- orianna: ok -->
Resource reads `verificationStatus` + `verificationReport` from the session doc and returns a structured response.
Managed agent can call this after a build to read the report and propose targeted config fixes.
