# QA enforcement and breakdown discipline — UI-involvement split + §QA Plan + §QA Tasks

## Context

PR #59 (`chore: xfail test skeletons — dashboard Phase 1 (TP1.T1–T7)`, merged 2026-04-25 at `13:34:52Z`) shipped the retro-dashboard Phase 1 ingest+render pipeline with this PR-body line:

> `QA-Waiver: non-UI, non-user-flow change — tools/retro pipeline only; no browser rendering or route changes. Akali Playwright flow not applicable.`

The waiver was false. The deliverable IS a static HTML dashboard authored to be opened in a browser and visually inspected — that is its sole purpose. Multi-layer process failure:

1. **Rule 16's `_User-flow_` definition is too narrow.** It enumerates "new routes / new forms / state-transition changes / auth flows / session lifecycle changes" — none of which describe a static-HTML dashboard generated by a CLI script. A read-only browser-rendered artifact slipped past every Rule-16 surface (the agent-def, the pre-PR Akali dispatch, the pr-lint workflow's keyword-match) because none of them named "browser-renderable output" as in scope.
2. **The PR's `Test plan` was fixture-only.** Seven xfail suites passed against fabricated fixtures. No suite exercised the pipeline against the user's real `~/.claude/projects/.../events.jsonl` — which is what determines whether a dashboard renders content or empty.
3. **The plan's DoD-(d) called for fixture-only smoke**; DoD-(e) cited a §4 real-data acceptance gate, but no committed test enforced it. So when reviewers ticked the DoD checkboxes, the gate was structurally invisible — the QA contract lived in prose, not in a verification command.
4. **The coordinator flipped the merge checkbox without running the artifact.** No human and no agent opened the rendered HTML against real data before merge.
5. **Reviewers (Senna + Lucian) approved.** They did not flag the QA-Waiver line because nothing in their plan-fidelity rubric required them to verify the waiver's accuracy against the deliverable's actual surface.

Net: the user installed the dashboard, opened it, and it was empty. Five guards were nominally in place; none caught the gap.

This is not a bug fix. It is a planning-discipline failure: at ADR time, no one declared what QA *means* for this deliverable; at breakdown time, no one converted that declaration into concrete tasks; at PR time, the waiver path absorbed the absence. **The fix is upstream — at planning and breakdown — not downstream at PR review.** Project agent-network-v1's DoD names "data transparency and accuracy" and "a dashboard for visibility into how well the system is performed" as primary outcomes; QA gaps that ship empty dashboards directly threaten that DoD.

This ADR proposes:

- **Plans (ADRs)** carry a mandatory `## QA Plan` section that explicitly declares whether the deliverable has UI involvement, and routes to one of two QA branches. There is **no third "skip QA" branch**.
- **Breakdowns** (Aphelios / Kayn task lists) carry a mandatory `### QA Tasks` subsection that converts the ADR's QA Plan into concrete enumerated tasks: which Playwright flows for UI; which fixture-vs-real-data smoke for non-UI; success thresholds; manual verification steps; xfail tests; PR markers.
- **Rule 16 in `CLAUDE.md`** is amended: `_User-flow_` becomes `_UI-involvement_`, defined to include any browser-renderable artifact. The blanket `QA-Waiver:` carve-out is removed; specific narrow waivers require explicit Duong sign-off.
- **Three lint surfaces** enforce the contract: `## QA Plan` at ADR promotion, `### QA Tasks` at breakdown commit, `QA-Verification:` PR-body marker on non-UI PRs.
- **Akali's agent-def** broadens scope from "user-flow routes" to "any browser-renderable output".
- **The ADR-template** in `_shared/` gains a §QA Plan stub so Azir / Swain / Karma include it by default.

The deliberate choice: enforce at the **earliest** seam (plan promotion) rather than the latest (PR merge). PR-time enforcement catches stragglers; plan-time enforcement makes QA *a designed property of the deliverable* rather than a post-hoc audit.

## Decision

### D1 — Two QA branches, no third (the core invariant)

Every ADR's `## QA Plan` answers exactly one question: **"Does this deliverable have UI involvement?"** — and routes to exactly one of two branches.

**UI-involvement branch.** Triggered by ANY of:
- Browser-renderable artifact: HTML, SVG, PDF for human visual inspection.
- Routes, forms, state transitions, auth flows, session lifecycle (preserves the original Rule 16 enumeration).
- CLI tools whose primary output is HTML/SVG/Markdown intended to be opened in a viewer.
- Dashboards, generated docs, status pages, reports — read-only or read-write, public or local-only.
- Anything where the user judges success by eyeballing a rendered surface.

Required QA: **Akali Playwright flow with video + screenshots**; Figma/wireframe diff if a design reference exists. PR carries `QA-Report: <path>`. **No `QA-Waiver:` permitted in this branch** — the only out is to demonstrate that none of the above triggers apply (which moves the deliverable to the non-UI branch).

**Non-UI branch.** Triggered by deliverables whose only outputs are:
- stdout / exit codes consumed by other tools.
- Files consumed by other tools (JSON, YAML, lockfiles, fixtures).
- Library APIs / internal endpoints with no human-facing surface.
- Build/CI/hook scripts whose output is logs.
- Pure refactors with zero observable delta.

Required QA: **Normal QA checks** as defined for *this specific deliverable* in the ADR's QA Plan (D2) and enumerated in the breakdown's §QA Tasks (D3). PR carries `QA-Verification: <commands-run-and-passing>`.

**No third branch.** The PR #59 path — "this is non-UI so we waive" — is closed. A non-UI deliverable still requires explicit Normal QA checks; it does not get to skip QA. The only universal skip is `QA-Waiver: <reason>` accompanied by a `Duong-Sign-Off: <timestamp>` line, which is treated as a one-off escape hatch, logged, and reviewed at the next weekend retro.

### D2 — §QA Plan section in every ADR

Every ADR proposed for promotion to `approved/` MUST contain a non-empty `## QA Plan` section. The section shape:

```
## QA Plan

**UI involvement:** yes | no

**If yes:**
- Surfaces in scope: <list of routes / pages / artifacts>
- Playwright flows: <list of user-stories Akali will exercise>
- Screenshots / video: <which screens, which states>
- Diff target: <Figma frame ID, wireframe path, or "no design reference">
- Success threshold: <pass/fail criteria; what counts as PARTIAL>
- Akali invocation pre-PR; report at `assessments/qa-reports/<slug>.md`; PR marker `QA-Report:`

**If no:**
- Output surface: <stdout / files / API / logs>
- Real-data acceptance gate: <command(s) that must pass against real inputs, not just fixtures>
- Fixture-vs-real split: <which tests fixture-only, which against real>
- Edge cases covered: <list>
- Manual verification: <coordinator steps before flipping DoD checkboxes>
- xfail tests: <test files / IDs>
- PR marker `QA-Verification: <commands>` citing the verification run
```

The linter (D7) checks for the `**UI involvement:**` line and at least one populated bullet under the chosen branch. Empty section or missing branch routing → reject.

`qa_plan: required` in plan frontmatter is the assertion that the section is populated. `qa_plan: none` is permitted only with a `qa_plan_none_justification:` string (sibling pattern to QA two-stage ADR's `qa_plan_none_justification` field) — this covers advisory ADRs whose downstream impl plans carry the actual QA Plan. Linter requires presence of one or the other.

### D3 — §QA Tasks subsection in every breakdown

Every `## Tasks` section authored by Aphelios or Kayn MUST contain a `### QA Tasks` subsection. The section converts the ADR's QA Plan from prose into enumerated, owner-tagged, estimate-tagged tasks. Examples:

**For UI deliverable:**
```
### QA Tasks

- T-QA1: Akali Playwright run on /dashboard happy-path (login → view chart → click filter)
  owner: akali
  estimate_minutes: 25
  artifact: assessments/qa-reports/<slug>-happy.md
  success: all 4 screens match Figma frame 12-7344 within agent-narrated diff tolerance
- T-QA2: Akali Playwright run on /dashboard error-path (network 500 from /api/data)
  owner: akali
  estimate_minutes: 15
  artifact: assessments/qa-reports/<slug>-error.md
- T-QA3: PR-body marker `QA-Report: assessments/qa-reports/<slug>-happy.md` (and -error if applicable)
```

**For non-UI deliverable (the PR #59 case, retroactively):**
```
### QA Tasks

- T-QA1: Real-data acceptance — run `node tools/retro/ingest.mjs --source ~/.claude/projects/<slug>/events.jsonl` and assert non-empty `events.json` output (>= 100 records on a typical user)
  owner: implementer
  estimate_minutes: 10
  success: ingest-emits-records command exits 0; output JSON has `events.length > 0`
- T-QA2: Render acceptance — run `node tools/retro/render.mjs && open dist/index.html`; coordinator visually confirms at least one card renders with non-zero data
  owner: coordinator
  estimate_minutes: 5
  success: dashboard non-empty on real user data
- T-QA3: Fixture-vs-real split — fixture suites stay in `tests/fixtures/`; real-data suite added under `tests/real/` runs only locally (CI uses fixtures); document the split in PR body
  owner: implementer
  estimate_minutes: 15
- T-QA4: PR-body marker `QA-Verification:` with the exact commands run and their pass-output (stdout snippet OK)
  owner: implementer
```

The linter (D7) checks for `### QA Tasks` heading inside any `## Tasks` block in a plan committed by Aphelios or Kayn (identity-gated). Missing → reject. Empty (heading present, no task lines) → reject. The breakdown agent's worktree-isolation rule already applies (Rule 20).

### D4 — Rule 16 amendment in `CLAUDE.md`

Replace the current Rule 16 text with the broader formulation:

> **16. Before opening any PR whose deliverable has UI involvement, Akali (`.claude/agents/akali.md`) must run the full Playwright MCP flow with video + screenshots and diff against the design reference (Figma frame, local wireframe, or coordinator-narrated reference) where one exists.** A deliverable has UI involvement if it produces ANY browser-renderable artifact intended for human visual inspection — including but not limited to: routes, forms, state-transition changes, auth flows, session lifecycle changes, static HTML pages, dashboards, generated reports, SVG/PDF artifacts, or CLI tools whose primary output is HTML/SVG/Markdown rendered for human eyes. Report lives under `assessments/qa-reports/` and is linked in the PR body via a `QA-Report: <path-or-url>` line. Enforced by `.github/workflows/pr-lint.yml` (PR body linter). **No blanket `QA-Waiver:` is accepted; specific narrow waivers require a `Duong-Sign-Off: <iso8601-timestamp>` line on the same PR body and are logged for weekend-retro review.** Non-UI PRs are exempt from `QA-Report:` but require `QA-Verification: <commands-and-results>` per the ADR's §QA Plan non-UI branch.

This decommissions the original `_User-flow_` enumeration as a closed set. The new wording reads `_UI-involvement_` as an open category with examples; ambiguous cases default to UI-involvement (fail-conservative).

### D5 — Plan-structure linter extension

Two lint surfaces, one new and one extended, both at pre-commit:

**Surface 1 (extension)** — `scripts/hooks/pre-commit-zz-plan-structure.sh` adds:
- Any plan path matching `plans/proposed/**.md` OR `plans/approved/**.md` MUST contain `## QA Plan` heading with non-empty body, OR have `qa_plan: none` + `qa_plan_none_justification: <string>` in frontmatter. Missing both → reject.
- The `## QA Plan` body MUST contain a line matching `^**UI involvement:** (yes|no)`. Missing → reject. (Case-insensitive on the value.)

**Surface 2 (new)** — `scripts/hooks/pre-commit-breakdown-qa-tasks.sh` (called from the same pre-commit chain) checks:
- If the committing agent identity is `aphelios` or `kayn` (resolved from `STRAWBERRY_AGENT` / `CLAUDE_AGENT_NAME` / framework `agent_type`, same as plan-lifecycle guard), AND
- The diff touches a plan with a `## Tasks` section, THEN
- The plan MUST contain a `### QA Tasks` subsection (anywhere — typically inside `## Tasks` or in a phase subsection) with at least one task line. Missing → reject.
- The reject message names the failure clearly and points the agent at the breakdown contract in `.claude/agents/aphelios.md`'s Hard Rules.

Together these two surfaces cover plan-time and breakdown-time. PR-time is covered by D7.

### D6 — Pre-dispatch gate (sibling to UX-Spec gate)

Either (a) extend `scripts/hooks/pretooluse-uxspec-gate.sh` to also enforce §QA Plan, or (b) author a sibling `scripts/hooks/pretooluse-qaplan-gate.sh`. Recommended: (b) sibling — keeps the UX-Spec hook readable, avoids tangling two distinct gate concerns. The new hook fires before any **impl-dispatch** agent (Jayce, Viktor, Seraphine, Soraka, Talon, Ekko, Vi, Rakan) runs against a plan, and blocks if the plan lacks `## QA Plan`. Bypass: same `qa_plan: none` + justification frontmatter, or `QA-Bypass: <reason>` in the dispatch description (Duong only).

This catches the case where the plan-structure linter was somehow not fired (e.g. plan was created manually before linter installed) but an impl agent is about to dispatch — defense in depth. Allow rate is high (impl agents only fire on already-promoted plans, which already passed Surface 1), so the gate's normal output is `ALLOW qa-plan-present`.

### D7 — PR-lint extension

Extend `.github/workflows/pr-lint.yml`'s `pr-lint` job:

1. UI-classification logic stays the same (path-glob + body-keyword); add the new categories from D1 (browser-renderable artifacts — extend the path-glob to include `tools/**/*.html`, `tools/**/render*.{mjs,js,ts}`, `**/dashboard/**`, etc., and add body keywords `dashboard | static html | rendered output | visual inspection`).
2. UI-involving PRs: keep `QA-Report:` requirement. **Tighten the `QA-Waiver:` path**: accept only if the PR body also contains a `Duong-Sign-Off: <iso8601>` line. Otherwise fail.
3. Non-UI PRs: NEW requirement — `QA-Verification: <commands-and-results>` line in PR body, OR `QA-Verification-Skipped: <reason> Duong-Sign-Off: <iso8601>` paired escape. Missing both → fail.
4. Failure message names the missing marker and points at Rule 16 + the ADR's QA Plan branch.

### D8 — ADR-template

Add a new file `.claude/agents/_shared/adr-template.md` (if not present) or amend the existing template that Azir / Swain / Karma reference. The template includes a §QA Plan stub matching D2's shape with both branches commented out so the ADR author selects one and populates it. ## Open Questions

1. **Single QA-Plan gate vs. sibling UX-Spec gate — merge or keep separate?**
   a: Keep separate (`pretooluse-uxspec-gate.sh` + `pretooluse-qaplan-gate.sh`). Cleaner single-responsibility per hook; easier to debug.
   b: Merge into one `pretooluse-plan-discipline-gate.sh` checking both §UX Spec (UI plans only) and §QA Plan (all plans).
   c: Skip the pre-dispatch gate entirely; rely on plan-structure linter at promotion time only.
   Pick: **a** (separate) — UX-Spec is UI-only; QA-Plan is all-plans; merging tangles two different scoping rules. The cost of two hooks is one extra `python3` invocation per dispatch (~5ms); not material.

2. **Non-UI PR `QA-Verification:` content — free-form vs structured?**
   a: Structured with required sub-fields (`Commands:`, `Pass-Output:`, `Real-Data-Source:`).
   b: Free-form single line; linter only checks presence and non-empty.
   c: Free-form but linter requires at least one `$` or `>` character (heuristic for "an actual command was cited").
   Pick: **b** (free-form non-empty) — Duong's principle is simple-yet-clean; structured sub-fields are easy to write but easy to fake; the real signal is whether the implementer can describe the verification in their own words. CI can tighten later.

3. **`Duong-Sign-Off:` format — iso8601 timestamp vs free-form?**
   a: Strict iso8601 (`2026-04-27T10:00:00Z`); linter regex-validates.
   b: Free-form ("Duong, Apr 27"); linter checks presence of the literal `Duong-Sign-Off:` line.
   c: Cryptographic — require commit signed by `harukainguyen1411` GPG key.
   Pick: **a** (iso8601) — keeps log-grep predictable for the weekend-retro review; cost to write is one date stamp; (c) is over-engineered for the threat model (Duong already gates merges from his admin identity).

4. **Should the new `QA-Verification:` requirement apply retroactively to in-flight non-UI PRs?**
   a: Yes — every open non-UI PR added after this ADR lands must add the marker before merge.
   b: No — apply only to PRs opened after the workflow change deploys; in-flight PRs grandfathered.
   c: Apply to PRs whose plan is in `approved/` after this ADR lands (carve via plan-creation date, not PR-creation date).
   Pick: **b** (forward-only) — minimum disruption, no scramble across in-flight branches; the population of in-flight non-UI PRs at any moment is small.

5. **Should `qa_plan: none` for advisory ADRs require a downstream-implementation plan reference?**
   a: Yes — `qa_plan_none_justification:` must include `downstream_plan: <path>` pointing at the plan that owns the actual QA. Linter validates the path exists (or is in `proposed/`).
   b: Yes — required prose justification, no path validation.
   c: No constraint beyond the justification string.
   Pick: **a** (path required + linted) — aligns with the project's "data transparency" focus; an advisory ADR that abdicates QA without naming who owns it is a structural hole. Path-validation is cheap.

6. **Akali scope statement — list of artifact types, or principle-based?**
   a: Enumerate explicitly (HTML, SVG, PDF, dashboards, generated docs, ...). Easy to lint, easy to game by adding a new artifact type the list missed.
   b: Principle-based ("any browser-renderable artifact intended for human visual inspection"). Forces case-by-case judgement; harder to lint.
   c: Both — enumerate as examples, principle as the authoritative test. ADR §D4 already does this.
   Pick: **c** (both) — already drafted this way in D4; preserves both the lintable list and the catch-all principle.

7. **Should the §QA Plan section be required for `karma` quick-lane plans too?**
   a: Yes — same shape, no exception for quick-lane.
   b: No — quick-lane plans are by definition trivial; require only a single-line `qa_plan: <yes|no> + <one-sentence-rationale>` in frontmatter.
   c: No — quick-lane plans are exempt; Karma's collapsed plan-build-test pass implicitly covers QA.
   Pick: **a** (yes, full shape) — PR #59 was a quick-shipping flow that skipped layered checks; the cost of a 5-line §QA Plan is negligible. This is exactly where the discipline matters most.

8. **Should we backfill §QA Plan on already-approved plans, or only enforce forward?**
   a: Backfill all `plans/approved/**.md` and `plans/in-progress/**.md` in a sweep before turning on the gate.
   b: Forward-only — gate triggers only on plans whose `created:` frontmatter is `>= 2026-04-27`.
   c: Backfill on first-touch — when an already-approved plan is next edited, the linter requires §QA Plan to be added.
   Pick: **b** (forward-only) — backfill is high-toil with low marginal value (most approved plans are already in-progress with implementation-time QA already underway). (c) creates surprise rejections on cosmetic edits.
