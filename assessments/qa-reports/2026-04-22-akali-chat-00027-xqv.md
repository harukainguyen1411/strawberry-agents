## Findings

### F-NEW-01 — MEDIUM: New Session UI form sends wrong field names → 422 from API

The UI's "+ New Session" dialog sends `{brand_name, insurance_line, market}` to `POST /session/new`, but the API schema requires `{brand, market}` (the `line` field does not exist in the endpoint's request model). Result: 422 validation error on every UI-triggered session creation attempt. The error is rendered as `[object Object]` in the form (JS error object not `.message`-extracted). Session creation only succeeds via raw API calls (confirmed by manually constructing the correct payload).

**Reproduction:** Click "+ New Session", fill any brand name + market, click "Create" → 422, form shows "[object Object]".

### F-NEW-02 — MEDIUM: F3 UI chat history not restored on page reload

The `/history` endpoint returns tool call history server-side (14 calls persisted). However, the chat-messages UI div is completely empty after page reload. The UI does not call `/history` on load to hydrate the conversation. The ConversationStore server-side fix (F3) works for API consumers, but the demo-studio front-end does not display previously executed tool calls or assistant messages when reopening a session.

**Evidence:** After reload, `document.querySelector('.chat-messages').innerHTML === ""` and body shows "Waiting for your first message..." despite 14 tool calls in /history.

### F-NEW-03 — LOW: Agent build-trigger loop on first configure turn

After completing 14 config-setting tool calls, the agent repeatedly called the build MCP tool, creating a user-facing "Deploy this demo?" confirmation loop. This is a behavioral issue with the agent's turn logic — it should wait for explicit user instruction to build rather than proactively looping on build calls. The session eventually ran up $0.0156 and remained in "configuring" status. Not a streaming fix regression, but a new issue.
