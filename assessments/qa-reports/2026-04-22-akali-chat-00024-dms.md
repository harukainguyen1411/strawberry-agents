## Findings

### F-01 — CRITICAL: `web_search_20241022` deprecated tool blocks all chat turns

Every call to POST /chat on any new session results in an immediate Anthropic 400:

```
vanilla_sse_generator: run_turn error: BadRequestError('Error code: 400 - {
  "error": {
    "type": "invalid_request_error",
    "message": "tools.4: Input tag 'web_search_20241022' found using 'type' does
    not match any of the expected tags: ..., 'web_search_20250305', 'web_search_20260209'"
  }
}')
```

Reproduced across 4 independent sessions (`4a119bae`, `fbb73f10`, and both prior Akali test sessions). The tool list passed to the Anthropic API still includes `web_search_20241022`, which is no longer accepted. The fix is to update the tool definition to `web_search_20250305` or `web_search_20260209`. This makes the entire chat surface non-functional.

**Affected:** All new sessions. 100% reproduction rate.
**Commits implicated:** This appears to predate `45702a8` (Viktor hotfix was for managed-agent/MCP paths; vanilla SSE path was not updated).

### F-02 — HIGH: Silent failure — no error feedback in chat UI

When a run_turn error occurs, the chat-messages div remains empty. The user sees "Waiting for your first message..." even after multiple send attempts. The input textarea and Send button re-enable normally, but there is no toast, inline error, or status message. Users have no way to know their messages failed.

**Evidence:** UI snapshot `qa-chat-00024-chat-silent-fail.png`. `chat-messages` innerHTML empty after 3 failed turns.

### F-03 — INFO: `managedSessionId` present in logs for pre-existing session `35af5b9d`

The session `35af5b9da1d64a168d56507dd5a5d733` (pre-existing, not created during QA) continues polling `GET /v1/sessions/sesn_011CaHrMiyxPp2xV22nQpKPY/events` at the Anthropic API. This session predates the Viktor hotfix scope and may be a legacy managed-session session. New sessions created during QA show no managed session polling — PASS for new sessions as scoped.
