## GATHER → GENERATE Flow Test (session fdb839ec)

Prompt sent: `"Generate a motor insurance demo for Aviva UK"`

| Step | Expected | Observed | Result |
|------|----------|----------|--------|
| GATHER: agent reads prompt | Extract brand/line/market | Page title changed to "Allianz" (not "Aviva") — brand parse mismatch OR session pre-seeded | FAIL (brand wrong) |
| S2 `get_schema` fires | `get_schema` tool call logged | 0 tool calls in `/history` despite $0.006 spend | UNVERIFIED |
| `web_search` fires | Hosted search call | 0 tool calls logged | UNVERIFIED |
| S2 `set_config` writes | Config written, `get_config` returns it | 0 tool calls logged; S2 UP but no write confirmed | UNVERIFIED |
| User approval → `trigger_factory` | S3 build kick + job ref | S3 DOWN — cannot proceed | BLOCKED |
| `get_last_verification` returns report | S4 response with report | S4 DOWN — cannot proceed | BLOCKED |

**Root cause of UNVERIFIED steps**: The SSE stream (`GET /session/{id}/stream`) returned `net::ERR_ABORTED` when the session auth cookie expired mid-stream (nonce-reuse redirect). Tool calls ran inside the agent process but were not recorded in the session history (`toolCalls: []` in `/history` endpoint). The $0.006 cost confirms LLM tokens were consumed, but the stream abort cut off the response before persistence.

**Brand mismatch**: The session page title showed "Allianz" despite the prompt being "Aviva UK". Investigation shows the preview iframe defaults to "Allianz" from the session template, and the `/history` API returned `brand: "Allianz"` — suggesting the agent parsed/stored the brand before the stream abort, possibly conflating the default template value rather than the user's input. This is a separate pre-existing issue unrelated to 00024-dms.

## `_map_error` Check (Viktor hotfix 097b0e1)

No raw `_map_error` strings or raw tool error payloads observed in browser network responses or console output. The console errors observed were:

- `TypeError: Cannot set properties of null (setting 'textContent')` — recurring every ~1s in the dashboard. This is a JS bug in the dashboard's status-update polling loop: it attempts to set `textContent` on a DOM element before it's mounted. **[SEV-2, dashboard only, not chat path]**
- CORS errors on `demo-factory`, `demo-preview`, `demo-verification` health probes — these surface as `{"error":"Failed to fetch"}` in the dashboard UI, not raw `_map_error` strings. The `_map_error` hotfix appears to be working as intended for these.
