# merge-commit-not-exempt-from-plan-lifecycle-guard

**Date:** 2026-04-26
**Discovered via:** PR #69 merge attempt blocked by `pretooluse-plan-lifecycle-guard.sh`

## What happened

PR #69 (Frontend-UX Stream B, UX Spec template + linter) was blocked at merge time because its merge commit message contained a plan-path token. The plan-lifecycle guard scans commit messages via the PreToolUse Bash hook and fires on any commit touching or referencing protected plan directories. Merge commits auto-generated by GitHub include the PR title and branch name in the commit message — and this PR's branch name contained a plan-path substring. The guard fired, blocked the merge, and the PR is now stuck.

## The gap

The guard was designed to block plan moves and writes. It was not designed to handle merge commits, which are structurally different: no files are written to the plan directories, but the commit message contains incidental plan-path references. No exemption for merge commits exists in the current implementation.

## Generalizable lesson

When writing a guard that scans commit messages for path tokens, always add an exemption for merge commits (`git log --merges` / `$GIT_MERGE_HEAD` presence check / `Merge pull request` prefix detection). Merge commits are structurally read-only relative to the plan directory — they do not introduce new plan files. Firing the guard on merge commits creates a hard block with no self-service recovery path (the committer cannot rewrite a merge commit's message without Duong-side action).

**Rule:** Plan-lifecycle guard must detect and exempt merge commits before running the path-token scan.
