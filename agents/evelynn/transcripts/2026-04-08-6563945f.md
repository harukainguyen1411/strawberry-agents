## The better architecture: Firebase + home box as queue worker

Flip the home box from "server" to **"worker"**. It only ever makes **outbound** connections — same traffic pattern as the Discord Gateway idea for the Discord plan. It doesn't listen on any port, tunneled or otherwise.

**Topology:**
- **Frontend**: Next.js deployed to **Firebase Hosting**. Sister bookmarks `bee.web.app` (or `bee.<duong-domain>` if you already have a Cloudflare/Google domain). Mobile-responsive for free.
- **Auth**: **Firebase Auth** (Google sign-in or email link). Kills the shared-password hack entirely. She gets a real account; you get proper session management.
- **Job queue + state**: **Firestore**. Sister submits a job → frontend writes `jobs/{jobId}` with `{status: queued, prompt, userId, createdAt}`. Frontend subscribes to the same doc for live status updates.
- **File I/O**: **Firebase Storage** (GCS bucket). Sister uploads `.docx` directly from browser to Storage with a signed URL. Home box downloads from Storage, processes, uploads the result, writes the Storage URL back to the Firestore job doc.
- **Home box worker**: NSSM-supervised Node/Python process on your Windows machine. Uses **Firebase Admin SDK** with a service account to snapshot-listen on `jobs` where `status == queued`. When a job appears, acquires the shared `~/.claude-runlock/claude.lock` (Bard's contract), runs `claude -p` locally, writes output to Storage + updates the Firestore doc. Loop.
