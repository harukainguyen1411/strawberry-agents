## The cascade — root cause

`.github/workflows/auto-rebase.yml` triggers on every push to main. It loops **every** open PR, rebases + force-pushes, which **synchronizes** each PR → retriggers ci.yml, e2e.yml, preview.yml, myapps-pr-preview.yml, myapps-test.yml, unit-tests.yml, lint-slugs.yml, pr-lint.yml, validate-scope.yml.

**31 open PRs × ~9 workflows = ~280 runs per merge.** That's your cascade.

Secondary issue: it uses `git rebase`, violating Rule 11 (always merge).

**Fix options:**

1. **Delete `auto-rebase.yml` entirely.** Simplest. Merging agent runs `gh pr update-branch` on the next-in-line PR only. Cascade goes from 31 PRs to 1.
2. **Label-gate it.** Only rebase PRs labeled `auto-rebase` (e.g. ready-to-merge queue). Blast radius shrinks to whatever's actively queued.
3. **GitHub merge queue.** Proper solution — PRs merge serially with staged CI. More setup, but correct long-term.

My recommendation: **option 1 now** (one-line PR to delete the workflow), then option 3 as a Heimerdinger plan for proper merge-queue adoption once CI is stable.
