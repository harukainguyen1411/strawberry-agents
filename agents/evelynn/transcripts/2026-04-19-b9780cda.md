## Duong — 2026-04-19T13:38:22Z

## Evelynn — 2026-04-19T13:38:45Z

Short version — IBKR's Client Portal API is nothing like T212's. No API key model.

**How it actually works:**
1. You run a **local Client Portal Gateway** (Java jar from IBKR) on a server/VM.
2. A human logs into it **via browser** (username + password + 2FA).
3. That login produces a **session cookie** in the gateway process.
4. You hit `https://localhost:5000/v1/api/...` and the gateway proxies to IBKR with your session.
5. You must call `/tickle` every ~5 min to keep the session alive. Hard max ~24h — then re-login.

**Implications for us:**
- No IP allowlist; auth is session-bound to a gateway process.
- Serverless doesn't work — Cloud Functions can't own a long-lived gateway + session.
- Requires a VM (bee-worker) running the gateway 24/7 with a live logged-in session, and something tickling it. Re-auth needs a browser interaction every day-ish.

That's why the portfolio plan defaults to **CSV import as the v0 IB path** — it dodges the whole mess. Live IB polling is realistically a v2+ problem.
