## What the industry actually does

The **pipeline** is the CI/CD system (GitHub Actions, GitLab CI, CircleCI, Cloud Build). The **scripts** are what the pipeline runs. Every serious setup has both — the question is only where the logic lives.

Two patterns exist:

**A. Logic in YAML.** GitHub Actions workflow has 40 lines of `run:` steps — install, test, decrypt, deploy. No scripts. Common in small projects and tutorials.

**B. Logic in scripts, YAML is a thin trigger.** Workflow is ~10 lines that call `./scripts/deploy.sh`. The script is the same thing a human runs locally. This is what mature teams converge on — Netflix, Shopify, Stripe, Airbnb all do variants. Terms you'll see: "pipeline as code", "makefile-driven CI", "taskfile/justfile".

Pattern B wins once you care about: running the exact same thing locally and in CI, keeping secrets out of YAML, swapping CI providers without rewriting, and debugging failures without pushing commits.

Pattern B — **with the CI half deferred**. Scripts now, GitHub Actions workflow later as a thin trigger. The design guarantees the workflow is ~10 lines when you want it.

Not yet. Honest take before we add it:

## The trap with auto-revert on Firebase Functions

Firebase doesn't have atomic "promote revision" like Cloud Run — a deploy *replaces* the function. "Revert" means **redeploy the previous tag**, which is another full deploy cycle (60–120s). During that window, the broken version is serving traffic. So "auto-revert" on Firebase is really "auto-redeploy-previous", not instant rollback.

That's still worth doing — but the design needs to be honest about what it buys you.

## What I'd add to the ADR

**Prod smoke test → auto-redeploy-previous on failure.** Specifically:

1. After prod deploy completes, `release.yml` runs a post-deploy smoke step:
   - `GET /version` returns expected `BEE_VERSION`.
   - A configurable checklist of HTTP assertions (healthz, one read, one auth-gated read).
   - Max ~30s total.

2. If smoke fails:
   - Workflow looks up the previous successful prod tag from `logs/deploy-audit.jsonl` (or GH Releases API).
   - Invokes `scripts/deploy.sh myapps-b31ea functions --ref <previous-tag>`.
   - Posts a GitHub issue + Discord alert: "bee-v0.3.0 smoke failed, reverted to bee-v0.2.1".
   - Audit log records both the failed deploy and the revert.

3. **Guardrails:**
   - Revert only runs if there IS a previous successful tag. First deploy ever = no revert, just fail loud.
   - Revert runs once, not in a loop. If the revert itself fails, workflow stops and pages you — never cascade.
   - Smoke test failures during *staging* deploy block prod (no revert needed — prod never got the bad version).

4. **What this doesn't give you:**
   - Zero-downtime rollback. Bad version serves traffic for 1–2 min until revert completes.
   - Protection against data-plane damage done during those 1–2 min (if v0.3.0 wrote bad data to Firestore, revert doesn't un-write it). This is why the ADR's no-rollback-for-data-migrations non-goal stays.
   - Protection against failures the smoke test doesn't catch (subtle logic bugs, race conditions). Auto-revert is a safety net for catastrophic failures, not a substitute for good tests.

## Alternative worth considering

**Don't auto-revert, page instead.** Smoke fails → open GitHub issue + Discord alert tagging you + leave prod broken pending human decision. Reason: auto-revert can hide problems (flaky smoke test → spurious revert → you never notice), and for a one-human team the cost of 5 minutes of downtime while you decide is usually fine.

## Evelynn — 2026-04-17T14:06:55Z

Only Bee. Monorepo ≠ monolithic deploy. Here's why and how:

## The isolation is enforced at three layers

**1. The deploy axis is `<project> + <surface>`, not "the repo".**

```
scripts/deploy.sh myapps-b31ea functions
```

That command targets: Firebase project `myapps-b31ea`, surface `functions`. It invokes `firebase deploy --only functions --project myapps-b31ea`. Firebase only knows about the code in `apps/functions/`. Code in `apps/haruka/` or `apps/whatever/` is never uploaded — the CLI doesn't even look at those directories.

**2. Each app has its own `firebase.json` / deploy target.**

Adding a second app means adding `apps/haruka/firebase.json` + `secrets/env/haruka-xyz.env.age`. A Bee deploy reads Bee's firebase.json. It's structurally incapable of touching Haruka.

**3. Release-please versions each app independently.**

```json
// release-please-config.json
{
  "packages": {
    "apps/functions": { "package-name": "bee" },
    "apps/haruka":    { "package-name": "haruka" }
  }
}
```

A `feat:` commit touching `apps/functions/**` bumps Bee only → tag `bee-v0.4.0` → `release.yml` sees the Bee tag → deploys Bee only. Haruka's version and its last-deployed revision stay untouched. Different apps can be at different versions simultaneously.

## The full isolation picture

```
commit touches apps/functions/**
         │
         ▼
release-please: "bee needs a bump"  (ignores apps/haruka/**)
         │
         ▼
tag: bee-v0.4.0                     (not haruka-v*)
         │
         ▼
release.yml matches bee-v* tag
         │
         ▼
deploy-staging job:
  scripts/deploy.sh myapps-b31ea-staging functions
    → firebase deploy --only functions --project myapps-b31ea-staging
    → uploads ONLY apps/functions/lib/
         │
         ▼
deploy-production job (after approval):
  scripts/deploy.sh myapps-b31ea functions
    → same, prod project
```

Haruka's code, Haruka's Firebase project, Haruka's audit-log lines — all unchanged. Bee's audit log gets two new entries (staging + prod). `firebase functions:log --project myapps-b31ea` shows only Bee.

## Edge cases where a Bee deploy *could* accidentally affect something else

Worth being explicit about, since monorepos do create these traps:

1. **Shared packages.** If `apps/functions/` imports from `packages/shared/`, a change to `packages/shared/` will ship inside the Bee bundle. It won't *deploy* Haruka, but it can create a version skew where Bee runs new shared code while Haruka still runs old shared code (until Haruka redeploys). Fix: release-please config marks shared packages as "bumps-all-consumers", and a shared-package change triggers release PRs for every app that depends on it.

2. **Shared Firebase project.** If you ever host two apps in the same Firebase project (e.g., Functions + Hosting for the same project), deploying with just `--only functions` is safe, but deploying without `--only` would touch both. The ADR's contract ("surface is always explicit") prevents this.

3. **Shared secrets file.** If Bee and Haruka somehow shared one `secrets/env/` file, a secret rotation would force both to redeploy. The ADR's "one env per project" rule prevents this.

## Short version

**Deploying Bee deploys Bee.** Other apps, other Firebase projects, other surfaces — untouched. The monorepo is a checkout convenience; the deploy boundary is per-app-per-surface-per-project, enforced by `firebase --only ... --project ...` and by release-please's per-package versioning.

The *benefit* of the monorepo is that Bee's shared code, shared tooling, shared scripts, and shared CI live in one place. The deploys stay isolated.
