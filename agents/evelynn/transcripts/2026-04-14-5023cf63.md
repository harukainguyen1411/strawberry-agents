## Evelynn — 2026-04-14T03:08:21Z

Good question. The mess comes from a few compounding causes:

1. **Worktrees are created but never cleaned up after merge.** When agents finish a PR and it gets merged, nobody goes back to `git worktree remove` the directory or delete the local branch. The worktrees just accumulate.

2. **Build artifacts aren't gitignored upfront.** Every time a new tool gets introduced (Turbo, Playwright, Firebase), its cache dirs start showing up in `git status` because nobody adds them to `.gitignore` on first use.

3. **Session close sometimes fails silently.** The Apr 13 session is proof — `/end-session` ran halfway, left staged files uncommitted, and nobody noticed until today.

4. **No periodic hygiene sweep.** There's no automated or scheduled check that flags stale worktrees, uncommitted files, or gitignore gaps.

**Fixes I'd recommend:**

- **Post-merge worktree cleanup** — add a step to the PR merge workflow (or to `/end-subagent-session`) that removes the worktree and local branch after a PR is merged. This is the biggest win.
- **Gitignore-on-first-use rule** — when any agent introduces a new tool that generates cache/build dirs, they must update `.gitignore` in the same commit.
- **Periodic git hygiene** — a scheduled check (weekly or session-start) that lists stale worktrees, uncommitted files, and gitignore gaps. Could be a heartbeat script addition or a cron trigger.

Want me to have Pyke write a plan for the post-merge cleanup automation? That alone would prevent 80% of the worktree accumulation.

```

```
