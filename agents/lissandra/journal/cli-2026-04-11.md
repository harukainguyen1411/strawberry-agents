# Lissandra — CLI Session Journal, 2026-04-11

Reviewed PR #89, the Windows push autodeploy feature. Node.js webhook receiver, PowerShell deploy scripts, NSSM install script.

The HMAC verification was done correctly — raw Buffer body, timingSafeEqual, proper prefix handling. That part I did not need to touch. The git pull strategy (ff-only) was also sound.

What I found worth noting:

The lock file pattern had a blind spot that I have seen before in other services. When the Node process crashes and NSSM restarts it, the lock file survives. Nobody checks whether the PID inside it is still alive. The service then permanently refuses to deploy. This is the kind of failure mode that is invisible in testing because the test never crashes the process mid-deploy — it's what happens at 3am when something goes wrong for unrelated reasons.

The DEPLOY_REPO_ROOT situation was more subtle. The env var is commented out in the template, meaning it defaults to `process.cwd()`. The install script sets NSSM AppDirectory to the app subdirectory. So `process.cwd()` at runtime resolves to the wrong directory, DEPLOY_SCRIPT points at a path that does not exist, and every deploy silently fails with a spawn error. The webhook still responds 200, the lock file gets cleaned up, the logs show "spawn error: ENOENT" — but nothing actually deployed. This is the kind of thing that passes a manual smoke test (because WEBHOOK_SECRET is checked at startup and forces you to think about env vars, creating false confidence that all env vars are handled) and then silently fails in production.

The ObjectName issue in the install script I flagged as MEDIUM because unattended installs would either stall waiting for a password prompt or misconfigure the service. That one at least fails loudly enough to notice.

The `detached: true` + `stdio: "pipe"` combination is a common misunderstanding. People see "detached" and assume the child is fully independent, but the pipe file descriptors hold the event loop. Not a correctness issue for a long-running service, but it means shutdown blocks until the child finishes. Worth knowing.

Three changes-requested items total. The two deployment-breaking ones (stale lock, wrong REPO_ROOT path) need to be fixed before merge. The ObjectName one should be documented at minimum.
