# PR Secret Scan Pattern

## What to check

When scanning a PR diff for secrets:
1. Use `gh pr diff <number> --repo <org/repo>` — output can be large (250KB+), save to file via tool output
2. grep for: `sk-ant-`, `xoxb-`, `AIza`, `AKIA`, `ghp_`, `ghs_`, `-----BEGIN`, `Bearer [token]`, and also bare `=` assignments that look like literal values (not `os.environ.get`)
3. Check `.env.example` additions — these are OK as long as values are empty/placeholder (`=` or `=...`)
4. Check `deploy/secrets.sh` — GCP project IDs in scripts are not secrets (they are infrastructure identifiers, not credentials)
5. `os.environ.get("KEY", "")` patterns are safe — they read from environment, not hardcoded

## AI authorship in commits

- `gh pr view <num> --repo <org/repo> --json commits --jq '.commits[] | {sha:.oid, headline:.messageHeadline, body:.messageBody}'` gets full commit data
- References to "Claude" as a product name (e.g. "Karabiner config for Claude Code", "claude-sonnet-4-6 model") in commit messages or file content are NOT AI authorship violations
- The rule only applies to Co-Authored-By lines, "Generated by Claude", "Written by Claude" authorship claims
- Commit subject "feat: add Karabiner volume key remap config for Claude Code" refers to the Claude Code tool as a product noun, not as an author — this is acceptable

## GCP project IDs

`mmpt-233505` is a GCP project ID (infrastructure identifier, not a secret). These are routinely committed in deploy scripts and are not sensitive.
