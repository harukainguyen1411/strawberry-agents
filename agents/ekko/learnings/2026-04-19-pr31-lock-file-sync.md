# 2026-04-19 — PR #31 CI fix: package-lock.json out of sync

## What happened

PR #31 (`feat/usage-dashboard-build-sh`) added `usage-dashboard@0.1.0` as an npm
workspace and listed `ccusage` as a dependency, but the `package-lock.json` was
committed without running `npm install` first. Both `Lint + Test + Build (affected)`
and `Firebase Hosting PR Preview` failed in ~7s at the `npm ci` step with:

```
npm error Missing: usage-dashboard@0.1.0 from lock file
npm error Missing: ccusage@0.8.0 from lock file
```

## Root cause

Classification: **env-config** — stale/missing lock file entries. Not a real
regression, not a flaky test, not a billing limit.

## Resolution

By the time investigation completed, agent Jhin had already pushed the fix as
commit `12a817a` (`chore: resolve merge conflict — keep Jhin fix`). This commit
merged the origin's diverged state and included the regenerated lock file.
New CI runs were already triggered (all `pending` at time of check).

## Key learning

When a new npm workspace or dependency is added to `package.json` without
re-running `npm install`, `npm ci` will fail immediately. The 7s failure window
is the giveaway: setup step fails before any real work starts.

Pattern: "Both A and B fail in <10s at the same step" = shared infrastructure
failure (usually install, credentials, or workspace config) rather than per-job
logic.
