# /pre-compact-save — Pre-Compact Memory Consolidation

## When to use

- Before running `/compact` on an Evelynn or Sona coordinator session
- When the PreCompact hook blocks `/compact` and asks you to consolidate first
- When you want a mid-session snapshot without ending the session

## When NOT to use

- Subagent sessions (out of scope — Lissandra only handles coordinator sessions)
- Actual session end (use `/end-session` instead — it produces the full transcript archive)
- Immediately after a recent consolidation (redundant; worst case it produces a duplicate)

## Arguments

`` (optional): `evelynn` or `sona` to override auto-detection. Empty → auto-detect from session greeting.

## Protocol

1. **Preflight.**
   - Read the session jsonl path: `~/.claude/projects/<project-slug>/<session-id>.jsonl`. Derive `session_id` and `transcript_path` from the current environment.
   - If `` is non-empty, use it as the coordinator. Otherwise detect:
     - Scan the first 3 user messages for `Hey Sona` (case-insensitive) → coordinator = `sona`
     - Otherwise coordinator = `evelynn` (repo default per CLAUDE.md Caller Routing)
   - Cross-check `[concern: work]` vs `[concern: personal]` tags on subagent prompts. Contradiction → refuse with a diagnostic.

2. **Spawn Lissandra** via the Agent tool with `subagent_type: "Lissandra"`. Note: Lissandra updates `open-threads.md` and regenerates `INDEX.md` as part of the coordinator shard write, same as `/end-session` Step 6b. Prompt shape:

   ```
   [concern: <personal|work>]

   Consolidate this coordinator session before /compact. You are impersonating <coordinator>.

   - session_id: <uuid>
   - transcript_path: <absolute path to jsonl>
   - coordinator: <evelynn|sona>

   Run the full protocol in your agent definition (handoff shard, session shard, conditional learnings, journal entry, optional transcript excerpt, commit, sentinel). Write to `agents/<coordinator>/...` directories only. Use the coordinator's first-person voice.

   Report back with: artifact paths, commit SHA, push status, skipped steps (if any), warnings.
   ```

3. **Write coordinator identity hint file.**
   - Determine the coordinator name: use `$CLAUDE_AGENT_NAME` if set and valid (`evelynn`/`sona`); otherwise use the coordinator name resolved in step 1.
   - Write one lowercase line (`evelynn` or `sona`) to `.coordinator-identity` at repo root (`git rev-parse --show-toplevel`). This hint is consumed by `scripts/hooks/sessionstart-coordinator-identity.sh` on `claude --resume` to pin the correct coordinator identity when env vars are not available.
   - Example: `printf 'sona\n' > "$(git rev-parse --show-toplevel)/.coordinator-identity"`

Two separate hook layers — easy to confuse.

- **`.claude/settings.json` hooks** fire on **Claude tool events** (Bash invocation, Edit, Agent dispatch, SessionStart). Claude Code is the trigger.
- **Git hooks** at `.git/hooks/<verb>` fire on **git operations** (commit, push) regardless of who invoked them. Git is the trigger.

`commit-msg-no-ai-coauthor.sh` is a **git** hook, so it lives in the git layer. The chain:

1. `scripts/install-hooks.sh` runs once at setup. It writes a thin dispatcher script to `.git/hooks/commit-msg`.
2. The dispatcher (380 bytes, you can `cat .git/hooks/commit-msg`) loops over every `scripts/hooks/commit-msg-*.sh` and runs each in order.
3. When **anything** runs `git commit` — Claude, you in a terminal, a subagent, CI, a worktree — git itself invokes `.git/hooks/commit-msg`, the dispatcher fires the AI-coauthor check, exits non-zero on match, commit aborts.

The reason it's git-side and not Claude-side: catching it at the git layer means it works regardless of caller. **Secretary MCP Suite** — Sona's "secretarial assistant" workflow. **The problem this PR solves**

1. **Plaintext-on-disk footprint.** Each MCP needs a `.env` file with raw credentials sitting unencrypted in the workspace tree. Eight MCPs = eight files of secrets in plaintext. One stray `git add .` and they're in a commit. One stray AI agent reading the directory and they're in a transcript.

2. **Inconsistent provenance.** Every MCP has its own ad-hoc `.env`-handling code in its `start.sh`. No shared template, no shared rotation story, no shared audit surface. When a credential rotates, you're hunting through eight different scripts to confirm propagation.

3. **No agent-friendly secret rotation.** If I need to give Ekko a new Slack token, the only path is "Duong, edit `mcps/slack/.env`." There's no mechanism for an agent to consume an updated secret without you in the loop.

**The fix: age-encrypted blobs + ephemeral-decrypt-exec pattern**

- `secrets/work/encrypted/<service>-<key>.age` — committed encrypted blobs in strawberry-agents (the agent control plane)
- `tools/decrypt.sh` — single chokepoint that reads ciphertext on stdin, decrypts using your age private key (kept locally, never committed), materializes a one-shot runtime env-file under `secrets/work/runtime/` (gitignored), then `exec`s the MCP with the env populated
- `start.sh` — every MCP's start script now follows the same canonical shape, reduced to: locate blob, pipe to `decrypt.sh`, exec the server

- **No plaintext credentials on disk in the work repo.** The work repo only ever has the encrypted blobs from the agent repo's perspective; the runtime env-file lives in tmpfs-or-gitignored space and dies with the process.
- **Single security boundary.** All decryption goes through `decrypt.sh`; one place to harden, audit, log, threat-model.

## SuperAdmin self-invite via existing flow — how it actually works

Concrete walkthrough using your actual case: you (a SuperAdmin in org `O_main`) want to give yourself a SuperAdmin account in org `O_other`. You'll need a second email address (e.g. `duong+other@example.com`) — the system has a self-block that keys on email match, so you can't use the same email as your existing account.

### Step 1 — Invite into `O_other` with a placeholder role

You hit the existing public API:

```http
POST /v3/orgs/O_other/invites
Authorization: Bearer <your existing-account JWT>
Content-Type: application/json

{
  "email": "duong+other@example.com",
  "role": "Editor"
}
```

What happens behind that call:
- Middleware `mw.CheckPermissionForOrg` runs against rego policy (`authz/rego/main.rego:21-40`). The policy maps this route to permission `manage:org` (`data.json:826-829`). That permission is held by both `SuperAdmin` and `OrgOwner` (`data.json:1221-1234`). You hold SuperAdmin in `O_main`, but the rego is keyed per-org, so you only pass this gate if you're a SuperAdmin or OrgOwner in **`O_other` specifically**. **This is the precondition you have to already satisfy** — you must already be SuperAdmin in `O_other` for the self-invite case, OR you're inviting someone else into an org you administer.
- Handler `invites.go:26-66` validates the requested role against the `InviteRoles` allowlist at `model/roles.go:39-44`, which contains exactly `[ProjectAdmin, Editor, Viewer, Overview]`. **`SuperAdmin` is deliberately excluded from this allowlist.** So we use `Editor` here as a placeholder — it's the lowest-friction stepping stone. The role doesn't need to be meaningful; we'll overwrite it in step 2.
- The handler signs a JWT containing the invite metadata (target email, target org, placeholder role) and ships it to the target email address as an accept-invite link.

### Step 1.5 — Accept the invite (target side)

You open `duong+other@example.com`'s inbox, click the accept link:

```http
GET /v3/resources/users/accept-invite?token=<JWT>
```

This:
- Validates the JWT.
- Creates the user account if it doesn't exist yet, **OR** if the email is already a known user, attaches the existing user_id to `O_other` with role=`Editor`.
- Inserts a row into `users_orgs` table: `(user_id, org_id=O_other, role=Editor)`.

After this step, `duong+other@example.com` is a real member of `O_other` with placeholder role Editor. They have an org membership row that we can mutate in step 2.

### Step 2 — Promote to SuperAdmin

Back to your existing-account JWT (or the new account's JWT, doesn't matter — the role-update handler doesn't require self):

```http
PUT /v3/orgs/O_other/user-role
Authorization: Bearer <SuperAdmin JWT for O_other>
Content-Type: application/json

{
  "user_id": "<the new user_id from step 1.5>",
  "role": "SuperAdmin"
}
```

What happens here:
- Same `mw.CheckPermissionForOrg` gate fires — caller must hold `manage:org` in `O_other`. (Same precondition as before.)
- Handler `orgs.go:278-348` validates the requested role against the `AvailableRoles` allowlist at `model/roles.go:31-38`. **This allowlist DOES include `SuperAdmin`.**
- Handler-level gates fire (`orgs.go:296-339`):
  - **Self-block at line 296-298:** caller cannot promote a user whose email matches the caller's email. *This is why you need the second email.* If you tried with the same email, this line would 403.
  - **Authority check at line 325:** caller must hold `OrgOwner` or `SuperAdmin` in this org (redundant with the rego gate but defense-in-depth).
  - **Membership precondition at line 337-339:** target must already be a member of this org. *This is why step 1's invite-and-accept matters — without the membership row, this 404s.*
- Persistence: handler runs `UPDATE users_orgs SET role = 'SuperAdmin' WHERE user_id = ? AND org_id = O_other` (`sqlstore/users.go:833-848`). One row, one column, atomic.

Done. `duong+other@example.com` is now SuperAdmin in `O_other`.

### Why the asymmetry exists (the "design intent" question)

The two-allowlist split looks accidental but makes sense:

- **Invite endpoint** sends an unauthenticated accept-link to an email address. Anyone with the link can become whatever role the inviter chose. Letting that role be SuperAdmin via a single email click would mean SuperAdmin promotions happen across an out-of-band channel (email) without a second authority check after the user authenticates. Excluding SuperAdmin from invite forces all SuperAdmin grants to go through the role-update endpoint, which requires an authenticated JWT round-trip.
- **Role-update endpoint** takes an authenticated JWT and operates on a user_id (not an email) that's already an org member. That's the "you've been authenticated, you're already in this org, now your role is being changed by another authenticated SuperAdmin" model. SuperAdmin is allowed here because the audit chain is solid: who promoted whom, when, in which org, against an existing member.

The PR I closed (#2108 / #2109) would have collapsed both steps into a single `POST /v3/superadmin/invite-user-to-org` route. That's UX-nicer but security-worse — it would have created a path where SuperAdmin grants happen via a single email click, bypassing the membership-precondition and authenticated-JWT pattern. Closing them was the right call.
