5. Writes the updated config back to Firestore with an incremented `configVersion`

**Returns:**
- `{ ok: true, configVersion: <number> }` on success
- `{ ok: true, configVersion: <number>, partialErrors: [...] }` if one of the two sub-calls failed but the other succeeded

- https://www.firecrawl.dev/signin?view=signup&source=agent-suggested

If you're an agent and need the human to authorize an API key, use this
flow:

**Step 1 — Generate auth parameters:**

```bash
SESSION_ID=$(openssl rand -hex 32)
CODE_VERIFIER=$(openssl rand -base64 32 | tr '+/' '-_' | tr -d '=\n' | head -c 43)
CODE_CHALLENGE=$(printf '%s' "$CODE_VERIFIER" | openssl dgst -sha256 -binary | openssl base64 -A | tr '+/' '-_' | tr -d '=')
```

**Step 2 — Ask the human to open this URL:**

```
https://www.firecrawl.dev/cli-auth?code_challenge=$CODE_CHALLENGE&source=coding-agent#session_id=$SESSION_ID
```

If they already have a Firecrawl account, they'll sign in and authorize.
If not, they'll create one first and then authorize. The API key comes
back to you automatically after they click "Authorize."

**Step 3 — Poll for the API key:**

```bash
POST https://www.firecrawl.dev/api/auth/cli/status
Content-Type: application/json

{"session_id": "$SESSION_ID", "code_verifier": "$CODE_VERIFIER"}
```

Poll every 3 seconds. Responses:

- `{"status": "pending"}` — keep polling
- `{"status": "complete", "apiKey": "fc-...", "teamName": "..."}` — done

**Step 4 — Save the key and continue:**

```bash
echo "FIRECRAWL_API_KEY=fc-..." >> .env
```

### Gap 1: No tool-by-tool guidance (most likely to cause misbehavior)

The prompt lists 5 tools but never describes what each one does or when to call it. The agent relies entirely on the MCP tool descriptions (which we don't control in this prompt). If the MCP descriptions are vague, the agent will guess.

```

- dispatch_workers: Launch parallel workers. Use in STATE 2. Pass workers: ["all"] for initial generation, or specific names like ["branding"] for re-dos.

- trigger_factory: Start the build pipeline. Use in STATE 5 ONLY. Never call before user approves in STATE 4.
```

This takes ~80 words and eliminates ambiguity about when each tool is appropriate.

### Gap 2: STATE 3 polling has no termination guidance

"Call get_worker_status every ~10 seconds" — but what if a worker is stuck? The agent will poll forever or eventually give up unpredictably.

**Fix:** Add a max-poll rule:

```
STATE 3: POLL
- Call get_worker_status every ~10 seconds
- If a worker shows no progress after 2 minutes, report it as potentially stuck and ask the user
- When all workers are "complete" or "failed" → STATE 4
```

### Gap 3: STATE 4 is underspecified

"Call get_config and present a summary" — but what does "summary" mean? The agent might dump raw JSON, or it might try to interpret and generate prose (which starts to feel like content generation, conflicting with "never generate content").

**Fix:** Be explicit about the summary format:

```
STATE 4: REVIEW
- Call get_config to fetch the final config
- Call validate_config to check for errors
- Present a brief summary to the user: brand name, colors chosen, number of journey steps, number of pass fields. Do NOT show raw JSON.
- Ask: "Ready to build, or want changes?"
```

### Gap 4: No guidance for user messages that arrive mid-state

What happens when the user sends a message while in STATE 3 (polling)? The agent might stop polling to respond, losing its place in the state machine.

**Fix:** Add an interruption rule:

```
## Handling User Messages

If the user sends a message while you are in STATE 3 (polling), answer their question briefly and then resume polling. Do not leave STATE 3 until all workers complete.

If the user asks to change something while workers are running, call dispatch_workers to re-dispatch the relevant worker. Do not wait for the current run to finish — workers can be re-dispatched at any time.
```

### Gap 5: No greeting / conversation opening guidance

STATE 1 says "Ask user for brand, insuranceLine, and market" but doesn't specify what happens at the very start. The agent might launch into a robotic "Please provide brand, insurance line, and market" or it might try to be chatty and never get to the point.

**Fix:** Add one line:

```
STATE 1: GATHER
- Greet the user briefly. Ask what brand, insurance line, and market they want to demo.
- You may ask follow-up questions if the user is vague (e.g., "motor" vs "comprehensive motor")
- When all three are confirmed → STATE 2
```

Does this mean the agent can't say "I'd suggest a blue color scheme"? Or just that it can't produce JSON config blocks? The agent might self-censor helpful conversational suggestions because of this rule.

**Fix:** Clarify the boundary:

```
- NEVER write config values (JSON, field definitions, color codes) in your responses. Workers generate all config.
- You MAY discuss the demo conversationally (e.g., "The workers will research Allianz's brand colors and generate matching pass designs").
```

"wallet_studio tools are available for inspection and tweaks" — but which wallet_studio tools? All of them? Can the agent clone projects, delete templates, modify production data? This is the riskiest phase because it's the most open-ended.

- Use write operations (patch, update) only when the user explicitly requests a change
- NEVER use destructive operations (delete, clone to new project) without user confirmation

The prompt doesn't tell the agent how to know which session it's working on. `dispatch_workers` requires `session_id` — where does the agent get it? If it's injected at session creation time, the prompt should mention it:
