PR #24 is the v2 demo studio — it doesn't contain a new `preview.html`. - Side-by-side chat/preview layout with responsive mobile tab switching — matches the spec exactly
- Correct design tokens: `--red`, `--sand`, `--dark`, `--teal`, etc. — fully consistent

## What v3 should adopt or improve

### 1. Chat message avatars / sender labels (spec calls for them, v3 skips them)
The spec shows `[Bot avatar] Bot` as a distinct label above bot messages. v3 uses only background color to distinguish sides. With long agent responses containing markdown, mixed tool indicators, and thinking bubbles all appearing on the left side, there's no clear visual anchor for "this is the agent speaking." Adding a small `Bot` label or avatar dot above bot bubbles would resolve ambiguity.

### 2. Tool activity accordion should collapse to a summary (spec: "Applied 2 changes")
The spec defined tool rows as collapsible accordions that collapse to a phrase like "Applied 2 changes." v3 shows each tool indicator inline as a persistent row, with a checkmark when done. They don't collapse. Over a long session, this creates a wall of completed tool rows. The fix: after all tools in a burst complete, collapse them into a single summary line like "3 changes applied" with expand toggle.

### 3. Empty preview state
When a session is new and no config exists yet, the iframe loads an empty preview. There's no loading skeleton, placeholder illustration, or helpful message visible to the user inside the iframe wrapper. A centered "Waiting for your first message..." state in the preview area (outside the iframe) would communicate status clearly.

### 4. Preview refresh feedback is easy to miss
The green border flash (`preview-refresh-flash`) is 0.3s and covers a large surface. It's subtle enough to be missed. The spec mentions "Preview refreshed" as a chat message after a tool call succeeds. v3 relies only on the border flash. Adding a brief system message in chat ("Preview updated") would give users confirmation without depending on them catching the visual flash.

### 5. Mobile: tab transition direction should follow content order
On mobile, switching from Chat to Preview animates with `translateX(10px)` for both panels — they both slide the same direction. Preview is logically "further right" than chat. The chat→preview transition should slide left-out/right-in; preview→chat should slide right-out/left-in. Currently both use the same direction, which feels undefined.

### 6. The preview tab bar (inside preview.html) uses a different font stack
`preview.html` uses `-apple-system, BlinkMacSystemFont, 'SF Pro Text'` while `studio.css` uses `'Helvetica Neue', Arial`. These render differently on Windows and Android. The studio shell and the preview iframe use different type systems, which creates a subtle disconnect when the two panels are side by side. Standardizing on the system-ui stack (`system-ui, -apple-system, sans-serif`) across both would unify them.

### 7. The stop button is visually disruptive
The stop button (`background: #b91c1c`) appears in a dedicated `.stop-bar` between the generate bar and the input row. It uses a hard red that's heavier than the brand `--red`. It should use `--red` with a border, not a filled block, to reduce visual alarm while still being actionable.

## Accessibility gaps

- **No focus styles** on tab buttons beyond the browser default — all custom buttons suppress outline without replacement
- **`aria-live="polite"` is on `.chat-messages`** — good. But tool indicator rows and thinking bubbles are appended to the same container without role annotation, which means screen readers may announce them mid-sentence when they appear during streaming
- **Color-only phase indication** — the phase dots use color alone (red = active, teal = done, grey = pending). Adding a shape or label change would be safer for color-blind users
- **Preview iframe has `title="Demo preview"`** — correct. But the sandbox attribute omits `allow-popups` which could silently break any links inside the preview that open modals or external URLs
- **Mobile tab bar buttons** have no `aria-selected` or `role="tab"` — they should use a proper tab pattern

- The `config-version` monospace display in the preview toolbar is useful for debugging but has no tooltip explaining what it is — a `title` attribute would help
