# 2026-04-19 — E2E CI performance audit

## Context
Evelynn asked for a baseline + ranked speed-up plan for the two Playwright
workflows in `harukainguyen1411/strawberry-app` (`e2e.yml`,
`myapps-test.yml`). Deliverable landed at
`plans/proposed/2026-04-19-e2e-ci-performance.md`.

## Facts worth remembering

- **Best real baseline was a failing run.** Every recent green
  `myapps-test.yml` run on `main` was a no-op skip (diff filter missed).
  Run `24620828113` (PR V0.10) was the only one where Playwright
  actually executed in the last ~20 runs. Use it, but annotate clearly
  that test-runtime numbers are inflated by retry storm on empty
  Firebase secrets.
- **Two workflow diff-gate pattern.** `myapps-test.yml` intentionally
  has no `paths:` filter so the required check always reports green —
  gate is in-workflow via `git diff`. This means `paths-ignore` cannot
  be added naively without breaking branch protection.
- **Playwright config lives at `apps/myapps/playwright.config.ts`.**
  `workers: 1` on CI is the single biggest perf leak. `retries: 2` is
  the second. webServer = `npm run build && vite preview` inline.
- **Browser cache key is lockfile-wide.** Any dependabot bump in
  `apps/myapps/package-lock.json` evicts the Playwright browsers
  cache even when `@playwright/test` didn't change. Key should be
  scoped to the Playwright version only.
- **`gh run view <id> --json jobs --jq '.jobs[] | .steps[] | ...'`**
  is the cleanest way to pull per-step timings. `--log` is verbose;
  filter aggressively.
