Design improvements to our agent communication system (agent-manager MCP). Two problems to solve:

1. **Rigid turn order** — Currently agents must speak in strict turn order. If one agent takes a long time on a task, the others sit idle waiting for their turn. We need a more flexible model where agents can continue working or contribute while waiting, without breaking conversation coherence.

2. **Dynamic participant addition** — There's no way to invite a new agent into an ongoing conversation. Example: Bard and Pyke are discussing something and realize they need Swain's opinion — they should be able to pull Swain in mid-conversation.

Note: `invite_to_conversation` already exists as a tool but may not be fully working (Syndra was supposed to get a design question about this last session but the message never arrived).

Review the current system at `mcps/agent-manager/server.py` and `agents/memory/agent-network.md`, then propose a design that solves both issues. Keep it practical — we're a small system, not building enterprise middleware.
