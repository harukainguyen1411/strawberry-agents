## Key learning — distinguishing legitimate xfail amendments from impl-driven test relaxation

The dispatch flagged two amendments to a Rakan-authored xfail (PR #79) as needing special scrutiny. Both turned out to be **bug fixes that reconcile the xfail with its own plan contract**, not relaxations to fit a buggy impl. Heuristics I used:

1. **Does the plan mandate the changed behavior?** Adding `delegate_health_flag` to ALLOWED — yes, T.P2.4 DoD-(c) mandates the column, and TP2.T3-C asserts it in the golden. The xfail's ALLOWED set was internally contradictory with its own assertions; the amendment reconciles, doesn't loosen.

2. **Does the amendment match a pre-documented contract elsewhere?** The `read_ndjson_auto` substitution pattern is explicitly documented in the SQL header AND mirrors what `duckdb-runner.mjs` already does in production. The original test invocation pattern (`duckdb -json '<eventsPath>' < SQL_PATH`) was simply wrong against the SQL's documented invocation contract.

3. **Does the amendment widen what passes, or narrow it?** Both amendments only change WHICH column is permitted (still a fixed allow-list, not a wildcard) and HOW the SQL is invoked (still asserting same column shape). Neither weakens the assertion surface.

If any one of these had been "no", I'd have requested changes. Pattern: speculative xfails authored before column finalization will sometimes need reconciliation; that's expected and OK as long as the reconciliation traces to plan DoD or pre-existing runtime contracts.
