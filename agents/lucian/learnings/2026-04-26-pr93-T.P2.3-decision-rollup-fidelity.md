## Key learnings for future T.P2.x reviews

### L1 — `existsSync`-gated xfails are clean Rule-12

Rakan's xfail tests in `__tests__/queries-decision-rollup.test.mjs` and `__tests__/decision-axes-parser.test.mjs` use `{ skip: !existsSync(IMPL_PATH) ? SKIP_REASON : false }` per `describe()` block. The impl arrival flips them automatically — no test-file edits, no commit-history rewriting, no race conditions. Future xfail authors should adopt this pattern uniformly (TP2.T2 is the cleanest example to date).

### L2 — Parser unit tests don't protect the live ingest path unless the source-reader CALLS the parser

This is the F1 finding's structural shape: `decision-axes.mjs::parseDecisionFrontmatter` correctly throws on bad axes/confidence (per TP2.T2-E/F), but `sources.mjs::scanDecisionLogs` rolls its own field extraction and silently `continue`s on bad input. The parser's invariants only fire for callers of the parser. Lesson: when reviewing future T.P2.x ingest extensions, always check whether the live source-reader actually invokes the typed parser, or whether it duplicates parsing logic with weaker invariants. Look for `knownConfidences = new Set([...])` declared-but-unused as a tell.

### L3 — Plan B §3.5 wording lags plan B §4 file-tree truth (Azir escalation)

Plan B §3.5 line 212 says decision logs live at `agents/<coord>/memory/decisions/*.md`. Plan B §4 (capture script line 246, rollup script line 259) shows the canonical layout is `agents/<coord>/memory/decisions/log/*.md`. PR #93 implemented against §4 (correctly). When future plans inherit §3.5 wording for file-list entries, the discrepancy will recur. Filed as `[escalate: azir]` for ADR amendment to tighten §3.5.

### L4 — `kind: '<noun>'` vs `kind: '<noun>-<source>'` convention is now ambiguous

T.P2.2 used `kind: 'feedback-entry'` (matching plan T.P2.1 line 292). PR #89 used `kind: 'turn'` (stream event, no log file). PR #93 used `kind: 'decision'` and defended via the PR #89 precedent — but conflated stream-event naming with log-event naming. Future kind reviews should classify the source category (stream vs log) and apply the convention accordingly. This needs Azir/Swain doctrine if a third log-source lands in Phase 3.
