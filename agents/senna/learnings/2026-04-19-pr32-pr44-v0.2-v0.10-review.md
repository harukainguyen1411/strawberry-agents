# 2026-04-19 — PR #32 (V0.2 auth+allowlist) and PR #44 (V0.10 base-currency) code reviews

## PR #32 — APPROVED

- beforeUserSignedIn (not beforeCreate) + per-invocation Firestore read is the right pattern: no cache-TOCTOU, and users removed from the allowlist can't ride stale UIDs.
- A.1.7 (`eventType` assertion) and A.1.8 (per-invocation read count) are worth keeping as regression guards against anyone "optimizing" with a cache later.
- Non-blocking UX gaps: `SignInCallbackView` hangs on a non-magic-link URL; `window.prompt ?? ''` fallback fails opaquely; no throttle on `sendSignInLink` (mild enumeration vector even with server-side allowlist).
- PII echo in `HttpsError` message is low-risk (caller's own email) but worth tightening for log hygiene.

## PR #44 — CHANGES REQUESTED

- Critical regression: `src/views/SignInView.vue` `handleSubmit` has the `sendSignInLink` call commented out, so the success banner is a lie.
- Two parallel `/sign-in` routes across PR #32 and #44 (different SignInView files) — merge-order coin flip. Explicit chaining needed.
- Firestore rule permissiveness: user-create checks `baseCurrency` field but doesn't constrain the key *set* — a client can include `isAdmin` etc. Use `hasOnly(...)`.
- Dead `_listenerActive` flag in `useAuth.ts` — flag is set after the check, so the guard does nothing (module-scope code runs exactly once anyway).

## Patterns for next time

- When two feature branches both touch auth/router surface, check explicitly for divergent view paths (`@/views/auth/Foo` vs `@/views/Foo`).
- Commented-out production-path calls with UI that claims success are a classic "works in isolation, broken at runtime" bug. Always read `handleSubmit`/`onClick` bodies to the end, not just the surrounding structure.
- Firestore `create` rules that check presence of a specific field but not the *shape* of the keys set are a common allowlist-bypass vector in per-user-doc designs.
