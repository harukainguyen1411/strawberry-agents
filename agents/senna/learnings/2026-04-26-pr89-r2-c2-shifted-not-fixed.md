## What happened

Round 1 surfaced fixture-vs-prod drift on three axes (C1: dispatch lacks coordinator+ts; C2: SQL filters phantom `kind: 'tool_call'`; C3: `read_ndjson_auto` schema-inference fragility). Viktor patched all three. C1 and C3 verified clean. **C2 had the rename applied but the fix did not actually unstick the headline metric.**

The architectural ask on C2 was *"do not introduce a new classification scheme; align SQL with production ingest format"*. Viktor changed `kind = 'tool_call'` → `kind = 'turn'` and updated all three fixture jsonl files to use `kind: 'turn'`. But the SQL `tool_counts` CTE has a *compound* WHERE clause:

```sql
WHERE kind IN ('turn', 'dispatch') AND coordinator IS NOT NULL
```

The fix changed the first half. The second half — `coordinator IS NOT NULL` — silently still excludes every turn event in production because `parseParentSession` and `parseSubagentSession` emit turn events without a `coordinator` field. After the C3 schema pin landed, those turns now correctly surface as `coordinator: NULL` and get filtered out. Headline metrics (inline_tool_calls, delegated_tool_calls, delegate_ratio) all read 0/NULL/no-data against real production data.

Reproduced E2E by building a minimal Claude-shaped session under /tmp, running `node ingest.mjs` from the PR head, and piping through coordinator-weekly.sql. Result: `{"inline_tool_calls":0,"delegated_tool_calls":0,"delegate_ratio":null,"delegate_health_flag":"no-data","dispatch_count":1}`. The single non-zero column is the C1 fix landing correctly; everything else is degenerate.

## The pattern — "fix the named line, not the contract"

Round 1 lesson said: *"diff the fixture against actual ingest output."* I missed that diffing the *single field* `kind` is insufficient — the WHERE clause has TWO predicates, both must agree with production shape. The new fixtures still pre-bake `coordinator: "evelynn"` on every turn, so the SQL passes against fixtures while failing against real ingest. **Same fixture-vs-prod drift class, one field over.**

The C2-named regression test (`regression-c1-c2-dispatch-count.test.mjs`) writes synthetic JSON.stringify events with `coordinator` baked in, pipes them through `runSqlAgainstEvents`, and asserts ratio is non-null. This is tautological: of course the SQL counts events with the field the WHERE clause requires when you write fixtures with that field. The contract being claimed — "real ingest → SQL → non-null ratio" — is not exercised. The C1 unit-test block IS the only one that runs `runIngest`, and it only inspects the dispatch event, never the turn events from the same ingest run, never runs the SQL against that ingest output.

## Lesson — compound WHERE clauses need per-predicate fixture audit

When a SQL WHERE clause has multiple predicates, fixture-vs-prod drift can hide in *any one of them*. The audit checklist:
1. List every column referenced in WHERE / ON / GROUP BY clauses.
2. For each column, grep the ingest emitter for that field name on each event kind being filtered.
3. If the emitter omits the field, schema pin returns NULL → filter excludes the row.
4. Fixture files that hand-write the field do not exercise this path.

This is the SQL-clause-level extension of the round-1 "diff fixture against ingest output" lesson.

## Lesson — regression tests must round-trip ingest, not pre-bake fixtures

When the bug is "fixture lies about production format", the regression test that closes the bug MUST start from source-shape inputs (the JSONL files Claude actually writes), call the actual ingest function, and run the SQL against THAT output. Pre-baked event fixtures that are themselves the bug shape can pass forever without ever exercising the production path.

The minimum E2E shape:
- Set up a tmpdir matching `~/.claude/projects/<slug>/<session>/{<session>.jsonl,subagents/agent-X.{jsonl,meta.json}}`
- Call `scanAllSources({...})` (or `node ingest.mjs ...`)
- Pipe the resulting events.jsonl through the SQL
- Assert non-null ratio + dispatch_count > 0

The fixture jsonl files (`coordinator-weekly-events.jsonl` etc.) should themselves be GENERATED by the same E2E setup so future drift auto-surfaces.

## Lesson — when re-reviewing, repro before relying on test count

"33 tests green" was the headline. Round-1 had 39/39 tests green and the bug was real. Same here. **Whenever a re-review is on a PR where round-1 found fixture-vs-prod drift, do the empirical ingest→SQL run before reading any new test code.** Test-suite green tells you the test suite is internally consistent; it tells you nothing about whether the contract is satisfied. The 30-second tmpdir + ingest + duckdb run is decisive in a way that re-reading test source is not.
