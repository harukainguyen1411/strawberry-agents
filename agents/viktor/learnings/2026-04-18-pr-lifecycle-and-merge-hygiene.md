description: Pitfalls from opening follow-up PRs on already-merged branches, xfail file naming, and test result churn
type: feedback

**PR tracking after merge:** When a PR merges and you push new commits to the same branch, those commits are NOT part of the merged PR — they go to a new PR. Always check `gh pr view <n> --json state` before assuming a PR is still open. Open a fresh PR explicitly for the fix.

**Why:** Caused multiple rounds of confusion where Jhin and Caitlyn were reviewing stale SHA references across #159/#180.

**How to apply:** After every push, verify `gh pr list --head <branch>` shows an open PR with the expected SHA. If not, open one.

---

**xfail file naming:** Files named `*.xfail.test.ts` should only keep that suffix while their contents are `it.fails()` xfails. Once implementation lands and tests flip to plain `test()`, rename to `*.test.ts`. Rule 12 TDD discipline is satisfied by commit order in git history, not by filename.

**Why:** Jhin would have flagged naming-vs-content mismatch as a review finding.

**How to apply:** On the same commit that flips xfails to passing tests, rename the file.

---

**test-results churn during merge:** `dashboards/server/.test-results/unit.json` is regenerated by the pre-commit hook on every commit, causing it to appear dirty again immediately after committing. This blocks `git merge`. Workaround: `git stash -- <file>`, merge, resolve the add/add conflict with `--theirs`, pop stash.

**Why:** The file is committed to track test history but regenerated on every hook run.

**How to apply:** When merging and this file blocks, use the stash pattern above. Long-term fix: add it to `.gitignore`.

---

**Cherry-pick over reimplementation:** When another agent (Vi) has already seeded xfails and implemented a task on a cluster branch, cherry-pick their commits rather than reimplementing. Preserves authorship, avoids duplicate work, keeps TDD chain authentic.

**Why:** Almost reimplemented F3 CORS from scratch before noticing Vi's commits on xfail-seed-cluster.

**How to apply:** Before starting any task, `git log --all --oneline --grep="<task-id>"` to check if work already exists.
